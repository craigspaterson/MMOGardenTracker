# Legacy System Discovery Checklist

This checklist guides the discovery phase before rewriting MMOGardenTracker. Work through each section to extract enough understanding of the current system to inform the PRD, architecture document, and work packages.

---

## 1. Product Purpose

- [ ] Summarise in one sentence what MMOGardenTracker does for an end user.
- [ ] Identify the primary MMO game(s) or game genres the tracker targets.
- [ ] Confirm whether the product is a personal tool, a team tool, or a public service.
- [ ] Document who the intended users are (casual players, hardcore farmers, guild officers, etc.).
- [ ] Note any stated long-term vision or roadmap in the README, comments, or commit history.

---

## 2. Core Entities

- [ ] List every domain entity found in `Domain/GT.Domain/` (e.g. Garden, Plot, Crop, Harvest, User).
- [ ] Document the properties and relationships of each entity.
- [ ] Identify which entities are persisted vs. transient.
- [ ] Note any value objects, enums, or constants that are part of the domain.
- [ ] Check `Common/` for shared types used across projects.

---

## 3. Current Features

- [ ] Review all API controllers in `WebService/GT.Web.Api/Controllers/` and list each endpoint group.
- [ ] For each controller, list the operations exposed (GET, POST, PUT, DELETE, PATCH).
- [ ] Review the Angular `Website/src/` for UI feature modules and components.
- [ ] Map each UI feature to its backing API endpoint(s).
- [ ] Check `postman/` for any Postman collections that document existing workflows.
- [ ] Note any features that appear partially implemented or stubbed.

---

## 4. Business Rules

- [ ] Extract validation rules from `FluentValidation` validators in the domain or API layer.
- [ ] Document any conditional logic in service classes or domain methods.
- [ ] Note authorisation rules: which actions require authentication, which are public.
- [ ] Look for any rate-limiting, quota, or game-specific constraints in code comments or configuration.
- [ ] Identify any data integrity rules enforced at the database layer (constraints, indexes).

---

## 5. Data Model and Persistence

- [ ] Locate the `DbContext` class(es) in `Domain/GT.Domain/` and list all `DbSet<T>` properties.
- [ ] Review EF Core migrations to understand the database schema history.
- [ ] Document the primary key strategy (identity, GUID, natural key).
- [ ] Note any soft-delete patterns (e.g. `IsDeleted`, `DeletedAt` columns).
- [ ] Record audit fields (e.g. `CreatedAt`, `UpdatedAt`, `CreatedBy`).
- [ ] Check `appsettings.json` and `appsettings.*.json` for connection string names and database provider.
- [ ] Confirm whether SQL Server is the active provider and whether PostgreSQL is intended for future use.
- [ ] Check `Database/` for any SQL scripts, seed data, or schema snapshots.

---

## 6. API Surface

- [ ] List all API versions defined via `Microsoft.AspNetCore.Mvc.Versioning`.
- [ ] Record the base URL structure (e.g. `/api/v1/...`).
- [ ] Review Swagger/OpenAPI definitions generated by Swashbuckle at runtime.
- [ ] Note any request/response DTOs defined in the Web API project.
- [ ] Identify AutoMapper profiles that map between domain entities and DTOs.
- [ ] Check for any authentication/authorisation middleware or attributes on controllers.
- [ ] Document CORS policy configuration in `Startup.cs`.

---

## 7. Frontend Behaviour

- [ ] List Angular feature modules under `Website/src/app/`.
- [ ] Identify routing configuration and the primary navigation structure.
- [ ] List Angular services that call the API and the endpoints they target.
- [ ] Note how the API base URL is configured (check `environment.ts` and `environment.prod.ts`).
- [ ] List all third-party UI libraries in use (Bootstrap 4, ngx-bootstrap, Font Awesome, etc.).
- [ ] Review `angular.json` for build targets, output paths, and any non-standard configuration.
- [ ] Note the TypeScript version and any notable `tsconfig` settings.

---

## 8. Non-Functional Requirements (Current State)

- [ ] Check `Startup.cs` or `Program.cs` for HTTPS redirection and HSTS settings.
- [ ] Review Serilog configuration (sinks, minimum level, structured logging fields).
- [ ] Check for any health-check endpoints.
- [ ] Note whether response caching or compression middleware is configured.
- [ ] Look for rate-limiting, request-size limits, or timeout configuration.
- [ ] Check `appsettings.json` for any environment-specific overrides.

---

## 9. Deployment and Runtime

- [ ] Determine whether there is a `Dockerfile`, `docker-compose.yml`, or any container configuration.
- [ ] Check `.github/` for CI/CD workflow files (GitHub Actions).
- [ ] Note the target runtime environment (IIS, self-hosted Kestrel, Azure App Service, etc.).
- [ ] Record how environment variables or secrets are injected at runtime.
- [ ] Check for any Azure, AWS, or other cloud-provider specific SDKs or configuration.
- [ ] Note the Angular build output folder and how it is served (static hosting, reverse proxy, etc.).

---

## 10. Quality and Testing

- [ ] List test projects under `Testing/` and identify their type (unit, integration, e2e).
- [ ] Note the test frameworks in use: MSTest, Moq, FluentAssertions, AutoFixture.
- [ ] Check code coverage setup (coverlet) and any coverage thresholds.
- [ ] Review `Website/src/` for Angular unit tests (Karma/Jasmine) and e2e tests (Protractor).
- [ ] Identify areas of the codebase with low or no test coverage.
- [ ] Note any known failing tests or skipped test categories.

---

## 11. Rewrite Decisions

- [ ] Confirm the target backend framework and language (e.g. .NET 8 or .NET 10 with ASP.NET Core).
- [ ] Confirm the target frontend framework and version (e.g. Angular 21).
- [ ] Decide whether to keep the existing SQL Server schema or redesign the data model.
- [ ] Decide whether to retain the existing API contracts or design new ones.
- [ ] Confirm authentication strategy for the rewrite (e.g. ASP.NET Core Identity, JWT, OAuth2).
- [ ] Decide on deployment target (containers, PaaS, static hosting + API).
- [ ] Agree on the definition of "done" for the rewrite: feature parity only, or new features included.
- [ ] Identify which existing tests will be ported vs. rewritten from scratch.
- [ ] Document any features that will be deliberately dropped or deferred.
